# Agent Development Guide — Orchestrator

## Service Overview

**Domain**: API Gateway / BFF (Backend for Frontend). CRITICAL infrastructure service that manages endpoint registry, proxy routing, manifest sync, UI app whitelisting, CSRF protection, security headers, and XSS guard.
**Service Name**: `Orchestrator`
**Risk Level**: CRITICAL — this IS the gateway. All UI traffic flows through this service. Changes require thorough review and testing.
**Database**: Uses SQL Server for endpoint registry, UI app registry, and caching.

### Architecture
This is NOT a typical microservice. Orchestrator is the single entry point for all client applications. It:
1. Receives all HTTP requests from UI apps
2. Resolves the target backend service from the endpoint registry
3. Proxies the request to the correct service
4. Applies security (CSRF, XSS guard, security headers, auth cookie resolution)
5. Returns the response to the client

### Key Hosting Components (in `Hosting/`)
- `OrchestratorHostingExtensions` — Registers all orchestrator services (`AddOrchestratorServices`, `UseOrchestratorPipeline`, `PrepareOrchestratorAsync`)
- `ProxyRequestInfo` — Request metadata for proxy routing
- `ProxyRequestXssGuard` / `ProxyRequestXssOptions` — XSS input validation for proxied requests
- `AuthCookieResolver` — Resolves auth cookies for BFF auth flow
- `OrchestratorAuthOptions` — Auth configuration for the gateway
- `UiAppSeeder` — Seeds UI app whitelist from configuration

### Key Hosting Middleware (in `Hosting/Middleware/`)
- `SecurityHeadersMiddleware` / `SecurityHeadersOptions` — Adds security headers (CSP, X-Frame-Options, etc.)

### Key Hosting Endpoints (in `Hosting/Endpoints/`)
- `ProxyEndpoints` — Core proxy routing endpoints
- `TechEndpointBase` — Base class for orchestrator endpoints
- Admin endpoints in `Hosting/Admin/`

### Key Services (in `Services/`)
- `EndpointRegistry` — Central registry of all backend service endpoints (populated from manifests)
- `EndpointRegistryServiceLocator` — Locates services by endpoint path
- `EndpointDescriptor` — Describes a registered endpoint
- `UiAppRegistry` / `UiAppDescriptor` / `UiAppMetadata` / `UiAppPayload` — UI app registration and metadata
- `ServiceUrlSelector` — Selects the correct service URL for proxy routing

### Services Subfolders
- `Services/Gateway/` — Gateway proxy logic
- `Services/Identity/` — Identity/auth resolution
- `Services/Import/` — Manifest import
- `Services/Manifest/` — Endpoint manifest sync (`EndpointManifest`)
- `Services/Parameters/` — Parameter forwarding
- `Services/Caching/` — Response caching
- `Services/TenantConfig/` — Tenant-specific configuration
- `Services/Tenants/` — Tenant resolution

### Key Models (in `Models/`)
- `EndpointEntry` — Registered endpoint with route, method, target service
- `EndpointOverride` — Per-tenant endpoint overrides
- `ServiceEntry` — Registered backend service
- `ServiceOverride` — Per-tenant service URL overrides
- `UiApp` / `UiAppEndpoint` / `UiAppOverride` — UI app whitelist entries
- `Models/Localization/` — Localization models

### Service-Specific Rules
- NEVER modify proxy routing logic without thorough testing — all UI traffic depends on it
- Endpoint registry is populated from service manifests (JSON files generated by `UpdateEndpointProxyFromRoutes` in each service)
- UI app whitelist controls which frontend apps can access the gateway — unauthorized apps are rejected
- CSRF protection uses double-submit cookie pattern — do not disable
- XSS guard (`ProxyRequestXssGuard`) validates request bodies on proxied requests
- Security headers middleware adds CSP, X-Frame-Options, etc. — do not weaken
- Auth cookie resolution handles BFF auth flow — converts cookies to JWT Bearer tokens for downstream services
- `PrepareOrchestratorAsync` runs at startup — loads manifests, seeds UI apps, populates endpoint registry
- Manifest sync happens periodically — services publish their endpoint manifests, orchestrator imports them
- The orchestrator uses `UseExceptionLogging()` and `UseRequestResponseLogging()` for observability
- Endpoint `[ContinuoProxyMethod]` attributes on downstream services control which endpoints are proxied
- HTML-accepting proxy endpoints require `[ContinuoProxyUiHtmlAttribute]` on the downstream service

## Platform Architecture & Conventions

### Service Bootstrap
- All services use `Bootstrap.CreateBuilder(args, serviceName)` and `Bootstrap.CreateApp(builder, serviceName)` from `Continuo.Observability`
- Target framework: .NET 10 (`net10.0`), nullable enabled, implicit usings enabled
- Services reference building blocks: `Continuo.Shared`, `Continuo.Observability`, `Continuo.Messaging`, `Continuo.Persistence`

### Database & Persistence
- Use `AddContinuoPersistence(config, serviceName)` for base DB context
- Use `AddServiceDbContext<TDbContext>(config, serviceName)` for service-specific DbContext
- Connection strings resolve via `PersistenceExtensions.ResolveConnectionString()` — in containers, security-api is checked first
- DB naming convention: `Continuo{ServiceName}Db` (e.g., `ContinuoOrderDb`)
- Primary keys: ULID string `nvarchar(26)` — never use auto-increment int or GUID
- Use `ContinuoDbContext` as base class for all DbContexts
- Outbox pattern: `dbo.OutboxMessages` table for reliable event publishing
- Always use `CreatedAtUtc` (DateTime, datetime2) for timestamps — UTC only
- SQL Server compatibility level 120 (no OPENJSON) — use `UseCompatibilityLevel(120)`
- Enable retry on failure: `sqlOptions.EnableRetryOnFailure()`
- Migrations: `MigrationRunner.ApplyMigrations<TDbContext>(app.Services, serviceName, ensureCreatedFallback: true)`
- ParameterDefinitions table: use `IParameterProvider` for configuration values

### Messaging (RabbitMQ + MassTransit)
- Use `AddContinuoMessaging(config, serviceName)` or inline `AddMassTransit` with `ConfigureRabbitMq(config, serviceName)`
- Event contracts live in `Continuo.Shared.Contracts` — reuse existing events before creating new ones
- Context propagation is automatic via MassTransit filters
- Outbox pattern ensures messages are published only after DB transaction commits
- Consumer naming: `{EventName}Consumer` in a `Consumers/` folder

### API Design
- Use Minimal API or Controllers — follow the existing pattern in the service
- All endpoints exposed via gateway must have `[ContinuoProxyMethod]` attribute
- HTML-accepting endpoints must have `[ContinuoProxyUiHtmlAttribute]`
- Call `app.UpdateEndpointProxyFromRoutes(serviceName, baseUrl: null, version: "v1")` before `app.Run()`
- Return `Result<T>` for operation results, `PagedResult<T>` for lists
- Use `Paging.NormalizePageSize()` for pagination
- Error responses: common `ErrorDto` / `ErrorResponse` model
- Health endpoint: `/health` and `/healthz` by Bootstrap

### Authentication & Authorization
- JWT Bearer auth configured via `JWT:SECRET`, `JWT:ISSUER`, `JWT:AUDIENCE`
- Tenant context: `AddTenantServices()` + `app.UseTenantMiddleware()` — `ITenantContext` via DI
- Context headers propagated: `X-Tenant-Slug`, `X-Correlation-Id`, `X-User-Id`, `X-Client-App`
- BFF auth: cookies are resolved to JWT tokens by `AuthCookieResolver`

### Service-to-Service Communication
- `ServiceCallExecutor` for inter-service HTTP calls (saga support)
- Named HttpClients via `AddAppCodeHttpClient("appCode", "serviceName")`
- Gateway proxy: `AddContinuoGatewayProxy(config)` — UI never calls services directly
- Service discovery: `AddContinuoServiceDiscovery(config, serviceName)`

### Observability
- Serilog (auto-configured), OpenTelemetry tracing, structured logging
- Error/Request logging to DB or file
- `UseExceptionLogging()` and `UseRequestResponseLogging()` explicitly enabled

### Code Quality & SOLID
- Single Responsibility: one bounded context per service
- Open/Closed: extend building blocks, don't modify base classes
- Interface Segregation: small, focused interfaces
- Dependency Inversion: always inject via DI, never `new` up services
- Keep controllers thin — logic in `Services/` folder
- Models in `Models/`, data in `Data/`
- Use records for DTOs and events

### Security Rules
- NEVER hardcode secrets — use config/env vars
- Validate all user input
- ALWAYS use parameterized queries or EF Core — never string concatenation in SQL
- XSS: `ProxyRequestXssGuard` validates proxied request bodies
- CSRF: double-submit cookie pattern — do NOT disable
- Security headers: CSP, X-Frame-Options, X-Content-Type-Options added by `SecurityHeadersMiddleware`
- Tenant isolation: ALWAYS filter by tenant context
- Auth endpoints are HIGH-RISK — require thorough testing

### Performance Rules
- async/await throughout — never `.Result` or `.Wait()`
- Use `Paging.NormalizePageSize()` — never unbounded result sets
- Keep DB transactions short, use outbox for async events
- Proxy routing must be fast — endpoint registry lookups should be O(1) or cached
- Response caching for static/semi-static responses

### Testing
- Test project: `Orchestrator.Api.Tests` under `tests/`
- `public partial class Program;` at end of Program.cs for integration tests
- Tests exist for: `ProxyRequestXssGuardTests`, `ProxySecurityHardeningTests`, `SecurityHeadersMiddlewareTests`
